# **Техническое задание (ТЗ)**  
**Разработка системы автоматизированного оценивания**

## **1. Введение**

### **1.1. Наименование системы**
Система автоматизированного оценивания учебных работ (далее — Система).

### **1.2. Краткое описание**
Система предназначена для автоматизации процесса создания, выполнения, сдачи и оценивания учебных работ с использованием критериального подхода. Она обеспечивает стандартизацию, объективность, прозрачность оценивания и улучшение обратной связи между преподавателями и студентами.

### **1.3. Цель разработки**
Создание веб-приложения с микросервисной архитектурой для автоматизации учебного процесса оценивания на основе заранее определённых критериев.

### **1.4. Целевая аудитория**
- Преподаватели
- Студенты

## **2. Требования к системе**

### **2.1. Функциональные требования**

| **ID** | **Функция** | **Описание** | **Приоритет** |
|--------|-------------|--------------|---------------|

| **FR-03** | Определение критериев оценки | Настройка критериев для каждого задания (бинарных или с градацией) | Высокий |

| **FR-05** | Критериальное оценивание | Автоматическая проверка по критериям с возможностью ручной корректировки | Высокий |
| **FR-06** | Алгоритм оценивания | Автоматическое определение оценки на основе матрицы критериев | Высокий |
| **FR-07** | Обратная связь | Комментарии преподавателя по критериям, возможность пересдачи | Высокий |

### **2.2. Нефункциональные требования**

| № | Требование | Значение |
|---|------------|----------|
| 1 | Производительность | Время обработки проверки работы ≤ 30 секунд |
| 2 | Доступность (uptime) | 99,5% в учебное время |
| 3 | Масштабируемость | Поддержка до 10 000 одновременных пользователей |
| 4 | Безопасность | Шифрование данных, разделение доступа по ролям, аутентификация через LDAP/OAuth |
| 5 | Надёжность | Резервное копирование, восстановление после сбоев |
| 6 | Удобство использования | Интуитивно понятный интерфейс, адаптивный дизайн |

## **3. Архитектура системы**

### **3.1. Архитектурный стиль**
- Микросервисная архитектура
- Событийно-ориентированная модель взаимодействия
- REST API для интеграций

### **3.2. Компоненты системы**
- **API Gateway** — единая точка входа
- **Сервисы:**
  - Auth Service (аутентификация)
  - User Service (пользователи)
  - Assignment Service (задания)
  - Grading Service (оценивание)
  - Storage Service (файлы)
  - Notification Service (уведомления)

### **3.3. База данных**
- **PostgreSQL** — основное хранилище
- **Redis** — кэширование
- **MinIO / AWS S3** — хранение файлов

### **3.4. Технологический стек**
- **Бэкенд:** Node.js или Python (Django/FastAPI)
- **Фронтенд:** React или Vue.js
- **Контейнеризация:** Docker, Kubernetes
- **Мониторинг:** Prometheus, Grafana

---

## **4. Алгоритм оценивания**

### **4.1. Логика проверки**
1. Проверка каждого критерия.
2. Подсчёт выполненных критериев.
3. Определение оценки по матрице:

| Критерии выполнены | Ошибки | Оценка |
|--------------------|--------|--------|
| Все                | Нет    | 5      |
| Все                | Небольшие | 4   |
| 1–2 не выполнены   | Значительные | 3 |
| Все не выполнены   | Ошибки по теме | 2 |

4. Возможность ручной корректировки преподавателем.

### **4.2. Бизнес-правила**
- Преподаватель может вернуть работу на доработку.
- Студент получает уведомления о статусе работы.
- Оценка может быть скорректирована преподавателем.
